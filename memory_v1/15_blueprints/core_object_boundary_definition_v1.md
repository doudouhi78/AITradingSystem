# 阶段3核心对象边界定义 v1

## 一、文件目的

本文件用于在模板正文编写之前，先定义阶段3四类核心对象的：
- 定位
- 职责
- 边界
- 输入输出关系
- 相互衔接方式

这份文件不直接充当模板，而是作为模板设计的上位依据。

## 二、四类核心对象总览

当前阶段3先定义 4 类核心对象：
1. 研究任务单
2. 候选交易方案卡
3. 样本拆解单
4. 审批提交包

它们在系统里的主链关系应当是：

样本拆解单
-> 研究任务单
-> 候选交易方案卡
-> 审批提交包

其中：
- 样本拆解单不是每次都必须存在，但在“借成熟样本启动”时非常关键
- 研究任务单是标准启动对象
- 候选交易方案卡是核心结果对象
- 审批提交包是最终交付对象

## 三、对象1：研究任务单

### 1. 定位
研究任务单是系统的标准输入对象。

它不是闲聊式提问，也不是执行过程记录，而是一张结构化任务委托单。

### 2. 核心职责
研究任务单负责定义：
- 研究对象
- 研究目标
- 研究边界
- 个人约束
- 输出要求
- 验证要求

它的任务是让系统知道“这次为什么启动、要产出什么、不能偏到哪里去”。

### 3. 边界
研究任务单负责“发起任务”，不负责：
- 展开完整方案细节
- 记录验证结果
- 代替审批结论
- 承担执行过程流水账

也就是说，它定义任务，不承载结果。

### 4. 输入来源
研究任务单可来自：
- 你的直接发起
- 样本拆解单转化而来
- 上一轮审批后的返修要求
- 某个自然变体研究需求

### 5. 输出去向
研究任务单的直接去向是：
- 进入研究执行流程
- 驱动候选方案草案生成

### 6. 完成标准
一张研究任务单如果满足以下条件，可视为合格：
- 能独立说明研究任务是什么
- 能约束 agent 不自由发散
- 能清楚指向目标产物
- 能支持进入下一步方案生成

## 四、对象2：候选交易方案卡

### 1. 定位
候选交易方案卡是系统的核心输出对象，也是最终审批的核心基础对象。

它不是某个单点信号说明，也不是单纯回测报告，而是一张完整候选方案卡。

### 2. 核心职责
候选交易方案卡负责承接：
- 机会来源
- 适用市场与标的范围
- 交易规则
- 仓位规则
- 风控规则
- 执行规则
- 验证摘要
- 适配性判断

它的任务是把研究结果组织成“可审批对象”。

### 3. 边界
候选交易方案卡负责“表达完整方案”，但不负责：
- 替代研究任务单定义输入边界
- 替代样本拆解单承接样本分析
- 替代审批提交包承接最终复审与建议结论汇总

也就是说，它是核心结果对象，但不是最终打包对象。

### 4. 输入来源
候选交易方案卡来源于：
- 研究任务单驱动下的方案生成
- 规则化结果
- 验证摘要回填
- 复审前的方案整理

### 5. 输出去向
候选交易方案卡的直接去向是：
- 被纳入审批提交包
- 被你最终审阅
- 后续在需要时被归档为样本或经验资产

### 6. 完成标准
一张候选交易方案卡如果满足以下条件，可视为合格：
- 不是灵感，而是完整方案
- 规则、风控、执行与验证信息基本闭合
- 可被复审和审批直接使用
- 不依赖额外口头解释才能成立

## 五、对象3：样本拆解单

### 1. 定位
样本拆解单是把成熟样本转化为系统资产的中间对象。

它的意义不在于直接审批，而在于把“外部样本”翻译成“内部可用对象”。

### 2. 核心职责
样本拆解单负责承接：
- 样本抓取的市场现象
- 样本完整规则结构
- 样本原始验证方式
- 样本可变体方向
- 对该样本的初步判断

它的任务是把“看过一个样本”变成“拥有一个可复用种子”。

### 3. 边界
样本拆解单负责“样本标准化”，但不负责：
- 替代研究任务单发起正式研究
- 替代候选交易方案卡表达最终候选方案
- 替代审批提交包进入最终审批

也就是说，它是前置转换对象，不是最终业务对象。

### 4. 输入来源
样本拆解单来源于：
- 成熟策略样本
- 前人经验方案
- 外部可解释研究案例

### 5. 输出去向
样本拆解单的直接去向是：
- 转化为研究任务单
- 进入样本资产库
- 为后续变体研究提供种子

### 6. 完成标准
一张样本拆解单如果满足以下条件，可视为合格：
- 能讲清样本抓什么
- 能压出完整结构
- 能支持转化为研究任务单
- 能支持后续自然变体研究

## 六、对象4：审批提交包

### 1. 定位
审批提交包是系统向你提交最终判断材料的统一打包对象。

它不是单独新发明的一套内容，而是对已有关键结果的统一组织与提交。

### 2. 核心职责
审批提交包负责汇总：
- 候选交易方案卡
- 验证摘要
- 复审意见
- 风险提示
- 建议结论

它的任务是让你在统一格式下完成审批，而不是自己去拼接多份材料。

### 3. 边界
审批提交包负责“最终提交”，但不负责：
- 替代候选交易方案卡表达方案本体
- 替代研究任务单定义研究目标
- 承担执行过程全部细节记录

也就是说，它是最后交付层，不是研究层对象。

### 4. 输入来源
审批提交包来源于：
- 候选交易方案卡
- 快速验证和二次验证摘要
- 复审工位输出
- 主脑汇总结论

### 5. 输出去向
审批提交包的直接去向是：
- 交由你审批
- 形成审批结论
- 后续决定进入淘汰、观察、模拟或候选执行

### 6. 完成标准
一份审批提交包如果满足以下条件，可视为合格：
- 审批所需材料已统一收口
- 不需要你自行拼接上下文
- 建议结论和风险提示明确
- 支持你直接做阶段判断

## 七、四类对象的边界关系

### 1. 研究任务单 vs 候选交易方案卡
区别：
- 研究任务单定义任务
- 候选交易方案卡表达结果

不能把结果内容提前塞进任务单，也不能让方案卡倒过来承担输入边界。

### 2. 样本拆解单 vs 研究任务单
区别：
- 样本拆解单服务“把样本变资产”
- 研究任务单服务“把任务送进系统”

样本拆解单可以转化为任务单，但两者不是同一个对象。

### 3. 候选交易方案卡 vs 审批提交包
区别：
- 候选交易方案卡是核心方案本体
- 审批提交包是最终交付包装层

审批提交包可以包含方案卡，但不能吞掉方案卡本身的独立性。

## 八、当前阶段结论

阶段3当前最先要稳定的，不是模板措辞，而是对象边界。

当前结论是：
- 研究任务单是标准输入对象
- 候选交易方案卡是核心输出对象
- 样本拆解单是样本转资产对象
- 审批提交包是最终审批交付对象

只有这四类对象边界稳定后，后续模板正文编写才不会混乱。
