# 研究任务单 v1 草稿

## 一、对象定位

这是一张用于启动一次正式研究任务的标准输入单。

它的目标不是聊天提问，而是把一次研究委托压成结构化任务对象，供主脑和后续执行工位稳定接手。

## 二、使用原则

- 一次任务单只服务一个明确研究主题
- 任务单负责定义输入边界，不负责承载研究结果
- 任务单内容必须足以约束 agent 不自由发散
- 任务单应优先服务“候选交易方案生产”，而不是开放式讨论

## 三、标准字段

### 1. 任务编号
用于标识本次任务，便于追踪、复盘和记忆沉淀。

建议写法：
- `RT-YYYYMMDD-01`

### 2. 任务名称
用一句简洁的话概括本次研究任务。

要求：
- 不写成模糊口号
- 直接体现研究方向或研究对象

### 3. 任务来源
说明这次任务从哪里来。

可选来源例如：
- 直接研究需求
- 样本拆解转化
- 变体研究
- 审批返修
- 市场观察触发

### 4. 研究对象
说明本次研究面向什么市场、标的池或资产范围。

例如：
- A股
- ETF
- 中证500成分
- 小市值股票池

### 5. 研究主题 / 市场现象
说明这次准备抓什么现象、逻辑或灵感来源。

例如：
- 趋势延续
- 超跌反弹
- 轮动效应
- 高换手溢价
- 某成熟样本的本土化变体

### 6. 研究目标
说明这次研究希望产出什么。

默认目标应写成：
- 产出完整候选交易方案

如有特殊要求，可补充：
- 做样本验证
- 做变体评估
- 做返修补全

### 7. 必须包含的模块
明确要求最终结果必须覆盖哪些模块。

建议默认包含：
- 机会来源
- 适用市场与标的范围
- 入场规则
- 出场规则
- 仓位规则
- 风控规则
- 执行规则
- 验证摘要
- 适配性判断

### 8. 个人约束
说明这次研究必须服从的个人现实条件。

例如：
- 只做日线
- 不做高频
- 不依赖盘中长时间盯盘
- 回撤不能过大
- 适合个人执行

### 9. 验证要求
说明本次任务至少要做到什么程度的验证。

例如：
- 先做快速验证
- 需要成本假设
- 需要看收益、回撤、年度分布
- 需要做基本稳健性检查

### 10. 输出要求
说明本次任务最后需要以什么形式交付。

建议默认写法：
- 输出候选交易方案卡
- 输出验证摘要
- 输出风险提示
- 输出建议结论

### 11. 当前阶段要求
说明这次任务允许推进到哪一步。

例如：
- 只产出候选草案
- 允许进入快速验证
- 允许进入完整方案补全
- 允许形成审批提交包

这个字段的作用是防止 agent 跑过头。

### 12. 备注 / 特别提醒
补充本次任务中特别需要注意的事项。

例如：
- 优先参考某成熟样本
- 不要偏离A股个人执行场景
- 本轮先粗筛，不追求深度优化

## 五、最小示例

```md
【研究任务单 v1】

1. 任务编号：RT-20260325-01
2. 任务名称：A股ETF趋势延续候选方案研究
3. 任务来源：直接研究需求
4. 研究对象：A股ETF
5. 研究主题 / 市场现象：趋势延续
6. 研究目标：产出完整候选交易方案
7. 必须包含的模块：机会来源、交易规则、仓位规则、风控规则、执行规则、验证摘要、适配性判断
8. 个人约束：只做日线；不做高频；适合个人执行；不依赖长时间盯盘
9. 验证要求：先做快速验证；考虑成本；至少看收益、回撤、年度分布
10. 输出要求：候选交易方案卡、验证摘要、风险提示、建议结论
11. 当前阶段要求：允许进入候选草案和快速验证，不进入真实执行
12. 备注 / 特别提醒：优先考虑规则清晰、可解释、可复现的方案
```

## 六、当前草稿判断

这版模板当前解决的是“任务如何被结构化发起”，还没有解决：
- 不同任务来源的字段差异化
- 模板版本升级规则
- 与样本拆解单的自动转换接口

这些内容留待后续阶段3继续细化。
