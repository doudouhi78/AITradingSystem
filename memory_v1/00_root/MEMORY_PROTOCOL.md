# 记忆协议

## 一、目的
这份协议定义：
- 什么信息应该进入项目记忆
- 信息应该进入哪一层
- 什么时候沉淀，什么时候清理

## 二、分层规则
### 00_root
放长期稳定的项目根信息：
- 愿景
- 总原则
- 长期边界

### 10_mainline
放当前主线判断：
- 当前阶段目标
- 当前系统状态判断
- 关键架构结论
- 阶段归档

### 15_blueprints
放项目蓝图资产：
- 项目总蓝图
- 阶段工程蓝图
- 关键结构蓝图

### 20_progress
放正在推进的工作草稿：
- 当下要做什么
- 还没正式定型的中间判断

特点：
- 可以频繁改
- 阶段结束后应清理/归档

### 40_experience_base
放经验和测试沉淀：
- lessons
- 样本
- 失败模式
- 阶段测试结论

### 70_runtime_roles
放角色层的运行记忆：
- seed
- config
- lessons
- samples
- working_memory

## 三、更新原则
1. 先进入草稿，再进入主线
2. 反复出现的经验，才能晋升为 lesson/sample
3. 长期记忆要薄，工作记忆要短
4. 不把长篇原始过程直接塞进长期记忆
5. 优先沉淀结论和样本，而不是沉淀长对话

## 四、角色记忆协议
- 角色共享真实项目现场，但 memory_root 应隔离
- 跨角色只传结构化 artifact，不传长篇原始 reasoning
- working_memory 用于当前任务，不用于长期堆积
- lessons 是压缩后的可复用经验，不是原始流水账

## 五、清理原则
- 阶段性工作完成后，20_progress 应归档后清空
- working_memory 在任务结束后应清理或重置
- promotion candidates 可以保留为候选，但不直接污染长期角色身份

## 六、项目记忆写入判定协议

### 1. 写入目的
项目记忆不是对话存档区，而是项目长期可复用认知的外部载体。

写入项目记忆的内容，必须至少满足以下目标之一：
- 能长期指导工程方向
- 能减少未来重复思考
- 能作为后续任务的稳定输入
- 能作为阶段接续时的关键上下文

### 2. 不直接写入的内容
以下内容默认不直接进入项目记忆：
- 长篇原始对话
- 临时 brainstorming 碎片
- 尚未收敛的随手判断
- 一次性执行细节
- 可从代码或文件系统直接重新获取的信息

这些内容应先停留在当前对话、草稿层或 working_memory 中。

### 3. 应优先写入的内容
以下内容在确认后应优先考虑写入项目记忆：
- 项目愿景、长期原则、长期边界
- 对项目终局形态的重要判断
- 当前阶段主线与阶段目标
- 经确认的系统蓝图、阶段蓝图、关键结构结论
- 经验证可复用的经验、lesson、sample
- 影响后续 agent 工作方式的制度、协议、接口约定

### 4. 写入前判断顺序
每当产生新内容时，按以下顺序判断：
1. 这是不是项目资产，而不只是当下聊天内容
2. 这是不是已经足够稳定，可以脱离上下文独立成立
3. 如果要写入，它应进入哪一层，而不是先问写不写得下

### 5. 分层落位规则
- `00_root`
  放长期稳定、跨阶段仍成立的内容，例如愿景、总原则、核心协议、终局蓝图
- `10_mainline`
  放当前阶段最关键的主线判断，例如阶段目标、当前系统状态判断、关键推进结论
- `15_blueprints`
  放项目级蓝图资产，例如项目总蓝图、阶段工程蓝图、关键结构蓝图
- `20_progress`
  放尚未定稿但值得保留的推进草稿、中间判断、阶段待办
- `40_experience_base`
  放已从真实推进中提炼出的 lesson、sample、失败模式、测试结论
- `70_runtime_roles`
  放角色运行需要的专属记忆，不替代项目级主线记忆

### 6. 默认工作原则
- 新内容先判断是否为项目资产
- 是项目资产，再判断应该进入哪一层
- 如果没有合适落点，先放入草稿层，不强行污染长期层
- 如果某类内容反复出现且长期有用，应考虑升级记忆结构

### 7. 结构演化原则
当前记忆结构以轻量可用为先，不追求一次性完美设计。

当出现以下情况时，可以对记忆结构做适应性改造：
- 某类高频重要内容长期没有合适落点
- 同类内容混放导致后续难以接续
- 当前分层无法支撑项目主线推进

记忆结构的改造目标是增强长期指引能力，而不是增加形式复杂度。
