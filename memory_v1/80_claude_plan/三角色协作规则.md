- **token 原则**：资源有限，不介入每个小细节，只在阶段切换/方向变化/需要拍板时出现
- **记忆位置**：`C:\Users\Administrator\.claude\projects\D--AITradingSystem\memory\`

### Codex-Builder（建设工程师）
- **Session ID**：`019d22ff-5216-75e2-869c-350e84020015`
- **调用**：`codex exec resume 019d22ff-5216-75e2-869c-350e84020015 --full-auto -o [结果文件] "任务"`
- **核心职责**：读真实代码现场、直接施工、自检、推进
- **不做**：不做最终验收，不拍板方向，遇真正岔路提 checkpoint

- **Session ID**：`019d37ef-f501-7a51-9fb9-333da32fa50d`

路径：`D:\AITradingSystem\coordination\`

| 文件 | 写入方 | 用途 |
|------|--------|------|
| `orchestrator_task.md` | Commander | 任务卡：目标/边界/输入/输出/验收标准 |
| `worker_result.md` | Builder | 结果卡：做了什么/改了哪些文件/结果/风险/下一步 |
| `builder_handover.md` | Builder | 项目状态交接，阶段性更新 |
| `reviewer_ready.md` | Reviewer | Reviewer 角色确认与独立判断 |

## Reviewer 验收标准（硬性）

只按"目标是否真实达成"验收：

1. **目标对齐**：直接服务当前阶段主线，不是系统扩张
2. **结果真实**：真实可运行、可复查，非空壳设计
3. **逻辑闭环**：输入-过程-输出必须闭合
4. **可重复**：同样输入能稳定重现主要结果
5. **不自欺**：严格区分"工程组织层完成"与"交易可信度建立"

**收口条件**：目标达成 + 结果可信 + 风险已明确，三项同时满足才 git commit。

## 协作哲学

- Builder 是核心生产力，不能被过度治理压死
- Reviewer 低频出现，否则系统被审查耗死
- Commander 管方向节奏，不下沉成实现者
- 角色之间只传结构化结论，不传长篇推理过程
- 三角色记忆隔离，不互相污染
