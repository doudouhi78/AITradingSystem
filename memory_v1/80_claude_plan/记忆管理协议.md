# 项目记忆管理协议
> Commander 负责执行，不可委托给 Builder 或 Reviewer。

建立：2026-03-29

---

## 一、记忆架构

```
memory_v1/
  00_root/          ← 长期稳定：愿景、原则、边界（极少改动）
  10_mainline/      ← 当前主线判断：阶段目标、关键架构结论
  15_blueprints/    ← 项目蓝图：字段规范、池子规则
  20_progress/      ← 进行中的草稿：任务卡、SOP、工作板
  40_experience_base/ ← 经验沉淀：踩坑记录、实验结论
  70_runtime_roles/ ← 角色记忆（各 Codex agent 的参考资料）
  80_claude_plan/   ← Commander 专属：计划、规则、执行状态
```

---

## 二、Commander 的记忆刷新职责

### 触发时机（以下任一即触发）
1. 一个 sprint 完成，Reviewer 给出通过并 commit
2. 阶段方向发生重大调整
3. 新工具/新实验正式纳入主线
4. 用户明确要求刷新

### 每次刷新必做

| 文件 | 操作 |
|------|------|
| `80_claude_plan/当前执行状态_YYYYMMDD.md` | 新建当日快照，更新六层进度、基线指标、工具栈 |
| `10_mainline/project_mainline_view.md` | 更新当前主线判断，版本号 +1 |
| `20_progress/working_draft_board.md` | 更新正在推进的工作项 |
| `40_experience_base/` | 沉淀本阶段踩坑和关键结论 |

### 刷新原则
- **只写事实，不写计划**：记忆里记的是"已发生的"，计划写在任务卡里
- **结论优先**：记结论和判断，不记推理过程
- **及时清理**：过期的草稿卡、已完成的工作项及时归档或删除
- **不重复**：同一信息不在多个地方维护，选一个权威位置

---

## 三、各角色读记忆的方式

| 角色 | 读哪里 | 用途 |
|------|--------|------|
| Commander | 全部 `memory_v1/` + 自己的 claude memory | 决策依据 |
| Builder | `80_claude_plan/` + `10_mainline/` + `15_blueprints/` | 理解当前任务背景 |
| Reviewer | `80_claude_plan/` + `10_mainline/` + `40_experience_base/` | 独立判断依据 |

Commander 在派任务时，会在任务卡里注入必要的上下文摘要，不要求 Builder/Reviewer 自行遍历 memory_v1。

---

## 四、记忆健康检查（每月一次）

- 检查 `00_root/` 内容是否仍然准确
- 清理 `20_progress/` 中已完成超过 30 天的草稿
- 检查 `40_experience_base/` 是否有未沉淀的重要经验
- 确认 `80_claude_plan/` 的执行状态快照是最新的
