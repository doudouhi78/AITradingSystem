# 二阶段：交易逻辑与工程对应关系

---

## 二阶段的核心问题

> **Phase 1 证明了策略在历史上"能赚钱"。但 Phase 1 的系统缺三样东西：**
>
> 1. **判断层缺失**：策略在任何市场环境下都无脑入场，没有"现在不是好时机"的判断
> 2. **仓位逻辑粗糙**：固定 50% 持仓，不随市场风险动态调整
> 3. **回测假设偏差**：Phase 5 仿真显示实际滑点显著高于回测假设的 0.05%
>
> 一个真正可以执行的系统，需要解决这三个问题。

---

## Module A：回测基础修正

### 这一模块解决的交易问题

> **你拿来做决策的历史结论，建立在错误的成本假设上。**

Phase 5 仿真结果：60 个交易日中有 43 天实际滑点超过 0.1%，而回测使用的是 0.05%。
这意味着所有 Phase 1 的 Sharpe、最大回撤数字都偏乐观，是用假数字做出来的结论。

**必须先修正再推进，否则 Module B/C 是在沙子上建楼。**

### 具体交易方法

修正方向只有一个：把回测成本假设调整到与现实匹配。

```
回测滑点：0.05% → 0.1%
影响范围：
  - 单次往返成本：约 0.36%（原约 0.26%）
  - 高频交易可行性下降
  - 年化收益预期下调
```

## Module B：Gate 系统移植

### 这一模块解决的交易问题

> **策略在任何市场环境下都无脑入场——熊市、高波动、市场极端异常时，信号照发，仓位照开。这是趋势跟踪策略的最大杀手。**

趋势跟踪策略在趋势清晰的市场里赚钱，在震荡市里被反复打止损消耗。
如果能在市场不适合趋势策略运行时关闭入场，整体表现会显著改善。

Gate 系统的作用不是预测市场方向，而是**判断当前市场环境是否适合本策略运作**。

### 具体交易方法

**Gate 系统来自 D:\TradingSystem，是已经写好的判断层，直接移植。**

| Gate 名称      | 判断什么                          | 触发条件（待回测调参）              |
| ------------ | ----------------------------- | ------------------------ |
| DrawdownGate | 当前策略组合回撤是否超过阈值                | 组合从高点回撤 > 12% → 暂停新入场    |
| TrendGate    | A 股大盘（沪深 300）是否处于下降趋势         | 300 指数在 120 日均线以下 → 关闭入场  |
| BreadthGate  | 市场上涨面是否过窄（多数股票在下跌）            | 上涨股 / 全部股 < 40% → 关闭入场   |
| VolGate      | 市场波动率是否处于异常高位（A 股无 VIX，用代理指标） | 近 20 日波动率 > 历史 80% 分位 → 限仓 |
| GateScheduler | 统一调度四个 Gate，输出最终 allowed/blocked | 任一 Gate 触发 → 阻断入场信号     |

**Gate 不修改出场逻辑。** 已持有的仓位按原始策略止损或出场，Gate 只影响"是否新开仓"。

## Module C：R-based 仓位与 ATR 止损

### 这一模块解决的交易问题

> **固定 50% 持仓不会随市场风险变化。低波动市场你拿太少，高波动市场你拿太多，每一笔交易对账户的实际伤害是不均匀的。**

R-based 仓位的核心逻辑：**每笔交易的最大风险敞口固定**，仓位大小由止损距离自动决定。

```
止损价    = 入场价 - ATR(14) × 2.0
风险金额  = 账户净值 × 1%（每笔最多亏总资产的 1%）
仓位手数  = floor(风险金额 / (入场价 - 止损价))
```

这样无论是高波动还是低波动的标的，每笔交易对账户的最大冲击都控制在 1% 以内。

**这个系统来自 D:\TradingSystem，已经写好，直接移植并适配。**

### ATR 止损的具体方法

ATR 使用 Wilder 平滑方式（不是 SMA）：

```
ATR_today = ATR_prev × (13/14) + TrueRange_today × (1/14)
止损价    = 入场价 - ATR × 2.0（倍数可调，默认 2.0）
```

这比固定百分比止损更合理：在高波动时自动放宽止损，避免被噪声触发；低波动时收紧止损，锁定利润。

## Module D：策略仪表盘

### 这一模块解决的交易问题

> **系统跑了半年，到底状态如何？没有标准化的展示，你只能靠手动查日志。**

这一层不增加任何交易能力，但让你能快速判断系统健康状态：

- 策略当前净值曲线 vs 回测预期
- 各 Gate 状态（哪个在阻断入场）
- 最近 N 笔交易的实际滑点 vs 假设滑点
- 滚动 Sharpe / 最大回撤实时更新

### 具体实现方式

**不引入新框架。** 用 QuantStats 已有的 HTML 报告能力 + 简单 Jinja2 模板，每天自动生成一个静态 HTML 文件，不需要数据库或后端服务。

### 责任归属

| ------------- | -------------- | ------- |
| 展示什么内容的优先级    | **你**          | —       |
| HTML 报告模板生成   | Builder        | Builder |
| 与现有日志格式的对接    | Builder        | Builder |

## 四个模块的执行顺序与依赖关系

```
Module A（回测基础修正）
    │ 必须先做，所有后续模块的结论基于修正后的回测
    ▼
Module B（Gate 系统移植）
    │ Gate 参数需要在修正后的回测数据上调优
    │
    ├── Module C（仓位与止损）可与 Module B 并行开发，但联合验收
    │
    ▼
Module D（策略仪表盘）
    │ 在 B+C 集成完成后，接入所有数据源生成报告
    ▼
    Phase 2 完整交付
```
