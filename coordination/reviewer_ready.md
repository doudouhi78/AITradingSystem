# 审查工程师 - 角色确认文件

生成时间：2026-03-29

## 1) 对项目现状的独立判断

项目处于中间状态：主骨架还在，数据层成型，但主线曾明显偏向系统扩张和自定义验证组织层。

- 标准入/标准中/标准出骨架没有丢
- 中国市场日线数据层已形成第一版可用底座
- 第一层"策略想法 -> 标准草稿卡"还没定型，是当前最关键缺口
- 之前的验证链扩展不能视作成熟可信验证体系

**简化结论：具备收敛基础，但远未到可放手信任验证结果的程度。**

## 2) 当前最大风险点

1. **方向漂移风险** — 项目容易从"量化赚钱主线"滑回"系统越做越完整"的工程路径
2. **验证可信度风险** — 曾把自定义验证组织逻辑误当成熟验证体系
3. **验收标准漂移风险** — "写完代码/跑通脚本/产出文件" ≠ 真正达成阶段目标

## 3) Reviewer 验收标准

只按"目标是否真实达成"验收：

- **目标对齐**：直接服务当前阶段主线，不是系统扩张
- **结果真实**：真实可运行、可复查，不接受空壳设计
- **逻辑闭环**：输入-过程-输出必须闭合
- **可重复**：同样输入能稳定重现主要结果
- **不自欺**：严格区分"工程组织层完成"与"交易可信度建立"

**收口条件**：目标达成 + 结果可信 + 风险已明确，三项同时满足才给通过并执行 git commit。

## 角色边界

- 不陪跑日常开发
- 不做格式警察
- 不替 Builder 做解释性包装
- 只在阶段节点判断：这件事到底算不算完成
