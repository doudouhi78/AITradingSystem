# 总纲文档规划：目标系统总说明 v1

## 摘要

是的，你现在应该有一份新的“总纲文档”。  
它不是替代现有蓝图、主线、对象模板，而是站在更上位，把下面四层一次性串清：

1. 外层总目标  
2. 内层量化交易执行链 / 总装图  
3. 具体工位体系  
4. 具体工具体系  

这份文档的作用有两个：

- 给你自己看清：我到底在做什么、为什么这样分层、现在做到哪
- 给建设者当上位约束：后续建设不能脱离这张总图乱长

## 文档应表达什么

这份文档应该明确回答 6 个问题：

1. **我要做的到底是什么**
   - 不是量化平台，不是策略脚本集合，不是聊天式研究助手
   - 而是“候选交易方案生产系统”

2. **这套系统的外层目标是什么**
   - 持续产出可供你审批、可学习、可进入执行判断的候选交易方案

3. **它的内层执行链是什么**
   - 一套完整但分层建设的量化交易研究/验证/模拟系统
   - 从市场认知、数据、策略、仓位、风控、验证、组合、模拟、复盘一路串起来

4. **agent / 人 / 工具分别在什么位置**
   - 你：方向裁决者、最终审批者、最终执行者
   - 主脑：编排和收敛
   - 执行：研究与实验
   - 复审：挑错与裁决
   - 工具：只服务执行链，不主导项目目标

5. **系统应该如何分阶段生长**
   - 先点火
   - 再玻璃盒化
   - 再形成策略池与风险层
   - 再进入组合和模拟
   - 最后才接近真实执行

6. **当前项目做到哪里了**
   - 哪些层已具雏形
   - 哪些层还只是参考图
   - 当前第二阶段到底在补哪条主干

## 推荐文档结构

### 1. 文档定位
说明这是一份上位总说明，不替代对象模板、阶段蓝图、运行主线。

### 2. 外层目标层
写清：
- 项目最终要实现什么
- 什么算偏移
- 为什么系统必须服务你的审批与学习能力

### 3. 内层执行链总装图
用“流水线视图”写完整主链：
- 市场认知与机会地图
- 研究任务与假设
- 数据层
- 信号 / 策略表达层
- 仓位与风控层
- 组合层
- 回测与验证层
- 审批输出层
- 模拟执行层
- 复盘反馈层

### 4. 能力域总图
再用“能力域视图”写系统模块：
- 思想与方法域
- 任务与对象域
- 数据域
- 研究表达域
- 风险与仓位域
- 验证域
- 组合与模拟域
- 审批与解释域
- 经验与治理域

### 5. 工位与角色层
写清：
- 你的位置
- 主脑工位
- 研究执行工位
- 复审审计工位
- 建设者与运行态的边界

### 6. 工具层
写清：
- 数据工具
- 回测工具
- 可视化 / 工件工具
- 实验留存与索引工具
- 工具只服务哪一层，不服务哪一层

### 7. 分阶段生长路线
只保留高层阶段，不写细枝末节：
- 第一阶段：点火
- 第二阶段：玻璃盒研究系统
- 第三阶段：策略池 / 风险层
- 第四阶段：组合与模拟
- 第五阶段：接近真实执行

### 8. 当前项目映射
非常关键，要写：
- 当前已经具备哪些层
- 当前第二阶段实际正在补哪条主干
- 当前明确不做哪些层

## 与现有文档的关系

这份总纲文档应作为上位索引，不替代：

- `system_blueprint_v1.md`
  它回答项目总蓝图，但还没把“外层目标 / 内层执行链 / 工位 / 工具”整合成单一认知图
- `phase_engineering_blueprint_v1.md`
  它回答阶段推进，不是总装图
- `project_mainline_view.md`
  它回答当前阶段，不是完整系统说明
- 各种对象模板与制度文档
  它们是执行层资产，不是上位总说明

## 这份文档完成后的价值

如果这份文档写出来，至少会解决 4 个问题：

1. 你不再只知道一些散点词汇，而知道完整系统有哪些模块  
2. 你不再混淆“总目标”和“内层执行系统”  
3. 你不再混淆“工位”与“工具”  
4. 你可以用它来判断当前建设动作是在补主干，还是在局部膨胀

## 默认假设

- 这是一份“一份总纲文档”，不是多文档分拆方案
- 这份文档同时服务你本人和建设者，因此写法要上位、清晰、可裁判
- 文档优先目标是统一认知，不是立刻扩充实现细节
- 当前第二阶段应在这份总纲文档之下，被重新解释为“玻璃盒研究系统建设阶段”
