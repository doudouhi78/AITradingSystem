**建议报告**

结合我前两轮判断，你这个项目在第二阶段“玻璃盒研究系统建设”里，最该补的不是更多策略，也不是更多角色，而是把研究链做成一套稳定、可追踪、可复现、可比较、可复审的研究操作系统。

你的目标不是做通用量化平台，而是做“候选交易方案生产系统”。所以选型必须围绕这条主线：外部能力要补你的工件链、验证链、复审链，而不是把项目带偏成大而全的平台。

**我对第二阶段的总判断**

当前最核心的建设目标，应压缩成 4 个能力：

1. 研究对象标准化  
任务单、规则表达、风险说明、复审结论都必须是结构化对象。

2. 实验可比较  
每次基线、变体、返修都能直接比较参数、指标、数据版本和结论。

3. 过程可观测  
不是只看最终文档，而是能看到研究链每一步做了什么、为什么打回、证据是什么。

4. 数据与结论可复现  
同一个实验对象，未来能重跑、能追溯、能解释，不依赖聊天上下文。

这 4 个能力里，外部方案不要重叠建设，所以我不建议同时上很多相近工具。下面是我压缩后的最优组合。

**最推荐的能力组合**

1. 实验追踪层：`MLflow`
这是我最推荐你第二阶段优先接入的外部能力。

原因：
- 它直接补“基线/变体/复审”比较能力
- 支持参数、指标、工件、标签、数据集元信息追踪
- 对你现有 `runtime/experiments` 目录形态兼容度高
- 很多新一代 AI/研究工具也在往它兼容，桥梁价值高

你这里不要再单独造一套复杂实验索引系统。  
第二阶段里，`MLflow` 最适合做研究实验总账本。

建议它负责：
- `task_id`
- `baseline_id`
- `variant_id`
- `dataset_version`
- `rules_version`
- `cost_assumption`
- `metrics_summary`
- `review_outcome`
- `decision_status`
- 工件路径链接

来源：
- [MLflow Tracking](https://mlflow.org/docs/latest/ml/tracking/)
- [MLflow Model Registry](https://mlflow.org/docs/latest/ml/model-registry/)

2. 内部能力总线：`MCP`
这是第二阶段最值得尽早吸收的新趋势，但要用得克制。

原因：
- 它非常适合把“数据查询、实验检索、模板读取、审批打包、记忆查找”封装成标准工具
- 能明显增强 agent 调用的规范性和可审计性
- 是近几个月增长很猛的一条基础设施线，适合你这种项目型系统

但这里不要追求“接一堆外部 MCP server”，而是自己做少量核心 server。

建议只做 3 到 5 个：
- `experiment_index_server`
- `research_artifact_server`
- `project_memory_server`
- `review_decision_server`
- `market_data_meta_server`

它负责“工具协议统一”，不负责实验追踪，也不负责工作流编排。这样就不会和 `MLflow` 重叠。

来源：
- [MCP Servers](https://github.com/modelcontextprotocol/servers)
- [MCP Python SDK](https://github.com/modelcontextprotocol/python-sdk)
- [MCP Python SDK Releases](https://github.com/modelcontextprotocol/python-sdk/releases)

3. 过程可观测层：`Phoenix` 思路优先，部署可后置
这是我最推荐你第二阶段补的第三个能力，但比 `MLflow` 和 `MCP` 稍后一点。

原因：
- 你第二阶段强调“玻璃盒”，而 Phoenix 这类 tracing/eval 工具天然适合把研究链打成 trace
- 它补的是“过程透明”，不是结果留存
- 非常适合记录 agent 的任务链、工具调用、复审节点、打回理由

它和 `MLflow` 不重叠：
- `MLflow` 记录实验结果和工件
- `Phoenix` 记录研究过程和 agent trace

如果第二阶段前半段不想多引入系统，可以先按 Phoenix / OpenTelemetry 的数据结构设计你自己的 trace 字段，后面再接入真实工具。

建议它负责：
- `research_trace_id`
- `task_span`
- `rule_expression_span`
- `validation_span`
- `review_span`
- `checkpoint_event`
- `return_reason`
- `evidence_links`

来源：
- [Phoenix Docs](https://arize.com/docs/phoenix)
- [Phoenix Releases](https://github.com/Arize-ai/phoenix/releases)

4. 输出结构化层：吸收 `PydanticAI / OpenAI Agents SDK / LangGraph` 的设计原则，不建议整套迁移
这部分我建议“借方法，不换底座”。

你项目已经有自己的引擎和角色体系，第二阶段不适合再迁 agent 框架。  
但你非常应该吸收这些新一代 agent 框架的共同做法：

- 输出必须 schema 化
- tool 调用必须类型化
- handoff 必须显式
- guardrail 必须存在
- state/checkpoint 必须显式

这里最适合你的不是“上哪个框架”，而是把这些原则落到你自己的项目对象上。

建议直接定义这些项目级 schema：
- `ResearchTask`
- `RuleSpec`
- `DataSnapshotSpec`
- `ValidationPack`
- `RiskPositionSpec`
- `ReviewMemo`
- `ApprovalSubmission`

这条能力负责“结构化对象”，不负责 tracing，也不负责实验追踪，所以也不会重叠。

来源：
- [openai-agents-python Releases](https://github.com/openai/openai-agents-python/releases)
- [openai-agents-js Releases](https://github.com/openai/openai-agents-js/releases)
- [LangGraph Releases](https://github.com/langchain-ai/langgraph/releases)
- [PydanticAI Releases](https://github.com/pydantic/pydantic-ai/releases)

**不建议现在主接入的方案**

1. `Qlib`
值得长期借鉴，但第二阶段不建议整体接入。  
它更像完整研究平台，会把你从“玻璃盒研究系统”带向“大型量化平台整合”。

来源：
- [Qlib Releases](https://github.com/microsoft/qlib/releases)

2. `Great Expectations`
方向是对的，但第二阶段不是最高优先级。  
建议你现在先定义“数据检查模板”，等数据层复杂度上来再正式接它。

来源：
- [Great Expectations](https://greatexpectations.io/)
- [GE Data Docs](https://docs.greatexpectations.io/docs/0.18/reference/learn/terms/data_docs/)

3. `Prefect / Dagster`
第二阶段先不要上重 orchestration。  
你现在更缺对象和链路，不缺调度平台。

来源：
- [Prefect Docs](https://www.prefect.io/docs)
- [Dagster Overview](https://docs.dagster.io/getting-started/what-why-dagster)

4. `DSPy`
很值得关注，但适合第二阶段后半段甚至第三阶段。  
它更偏“优化提示程序和评估”，前提是你的工件链先稳定。

来源：
- [DSPy Releases](https://github.com/stanfordnlp/dspy/releases)

5. `NautilusTrader / FinRL / FinRobot / FinGPT`
这些都有参考价值，但对你第二阶段主干不是最短路径。
- `NautilusTrader` 更偏第三、四阶段的模拟与接近执行
- `FinRL/FinRobot/FinGPT` 更偏 AI-for-finance 平台与智能体研究，不是你当前最缺的主链能力

来源：
- [NautilusTrader Releases](https://github.com/nautechsystems/nautilus_trader/releases)
- [FinRL Releases](https://github.com/AI4Finance-Foundation/FinRL/releases)
- [AI4Finance Foundation](https://github.com/AI4Finance-Foundation)

**最适合你项目目标的第二阶段最小组合**

如果只保留最推荐、且能力不重叠的组合，我给你的最终建议是：

- `MLflow`
职责：实验总账本、基线变体比较、结果/工件追踪

- `MCP`
职责：项目内部能力总线，规范 agent 如何访问实验、记忆、模板、索引

- `Phoenix` 或 Phoenix-style tracing
职责：研究过程可观测、打回路径、证据链留痕

- `Schema-first project objects`
职责：统一任务单、规则表达、验证包、风险说明、复审、审批提交物

这 4 个东西各补一条主干，不重叠，而且正对你第二阶段目标。

**优先级排序**

1. 先做 `Schema-first project objects`
这是根，不先做这个，后面所有接入都会发散。

2. 再接 `MLflow`
先把实验比较体系立起来。

3. 再做 `MCP`
把项目能力标准化，避免 agent 直接乱碰文件和上下文。

4. 最后补 `Phoenix-style tracing`
把过程玻璃盒化，而不只是结果玻璃盒化。

**一句话结论**

你第二阶段最优路线，不是“上一个更大的量化框架”，而是把项目升级成：

`结构化研究对象 + 可比较实验系统 + 标准工具总线 + 可观测研究过程`

如果你要，我下一步可以直接把这份报告进一步收敛成一版“第二阶段落地实施清单”，按你项目目录结构写成非常具体的接入方案。
