# Agent 工具包使用说明

## 一、说明书目的

这份文档说明：

- 这个新项目如何使用复制过来的 agent 发动机
- 我们目前多 agent 协作中，哪些经验已经验证过，值得直接带过来
- 哪些治理不要一开始做得太重

目标不是把新项目直接变成旧项目翻版，而是让它在一个更干净的起点上展开。

## 二、当前已经带入的新项目能力

- `src/ai_dev_os/`
  - 可复用 agent 发动机代码
- `config/agents.json`
  - 角色配置模板
- `memory/roles/trading_zone/`
  - 新项目的角色记忆包
- `runtime/`
  - 工作区和角色记忆根目录

## 三、当前 starter 角色

### market_architect

负责：

- 把交易系统目标展开成第一阶段蓝图
- 澄清假设、约束、优先级和风险边界

### strategy_operator

负责：

- 承接清晰 brief
- 输出具体研究/实现结果
- 遇到不该硬猜的问题时主动 checkpoint

## 四、我们已经验证过的多 agent 协作经验

### 1. 核心执行角色不能被治理压死

我们已经验证过：

- 如果把主手强行变成 planning-only，会明显变慢变弱
- direct execution 往往比“先抽象成 rigid 计划”更自然

实用原则：

- 能在闭环内直接做完的小任务，就让主手直接做
- 真遇到方向岔路、前提失效、阶段边界时，再 checkpoint

### 2. 审查要看目标和真实现场，不要当字眼警察

我们已经验证过：

- 如果审查层只盯证据字段是否完美，会把系统耗死
- 审查更应该看：
  - 目标是否达成
  - 真实现场是否支持这个判断
  - 关键链路是否正确

实用原则：

- 证据只要求“足够判断”，不要求绝对完美
- 如果目标已达成且现场可判定，可以带着 `evidence_debt` 通过
- 只有真正不可判定、明显错误、或高风险边界不清时，才该打回或升级

### 3. 跨角色传递必须结构化，但不用追求绝对无损

我们已经验证过：

- 原始 reasoning 泄露会污染角色边界
- 但如果要求每次 handoff 都绝对完美无损，系统会卡死

实用原则：

- 角色之间只传结构化 artifact
- 下游角色可以结合真实工作区做独立判断
- 以“足够判断”为标准，不以“完美证据包”为标准

### 4. 共享真实现场，隔离角色记忆

实践证明，这个方向是对的：

- 多角色共享同一真实工作区
- 各自 memory_root 独立
- 角色训练样板独立

实用原则：

- 角色可以看到同一份真实现场
- 但不要共享长篇原始内部推理

### 5. 每个关键角色都应该有三通道

这是目前最有效的结构之一。

正常通道：

- 干活，产出预期 artifact

拒绝通道：

- 输入不合格时，结构化回吐问题

升级通道：

- 遇到不该继续硬猜的情形时，结构化上浮 checkpoint / escalation

这能有效避免把“上游输入问题”误判成“角色能力失败”。

## 五、建议使用方式

### Phase 0：先不要做重流程

起步阶段建议：

1. 人给出初始项目目标
2. `market_architect` 产出第一阶段蓝图或 phase brief
3. `strategy_operator` 执行一个明确的子问题
4. 人根据真实结果决定下一步怎么扩张

### Phase 1：只在真实需要出现时再加流程

不要一开始就复制完整治理结构。
只有当以下问题反复出现时，才考虑加更多节点：

- 工作项跨时间越来越难追踪
- 执行质量需要独立审查
- 结果需要稳定归档和记忆回流
- 多个并行工作流已经开始出现

## 六、什么时候扩角色

### 增加 reviewer 型角色，当：

- 执行结果经常需要独立判断
- 风险边界越来越重要
- 自检已经不够稳

### 增加 recorder 型角色，当：

- 项目历史越来越难审计
- lessons / 决策记录需要自动结构化
- 阶段收尾和经验晋升需要更稳定机制

### 增加并行子组，当：

- 模块之间边界清楚
- 顺序推进已经成为真正瓶颈
- 合并成本仍可控

## 七、记忆机制建议

当前复制过来的引擎已经有一条比较合理的方向：

- 长期角色身份要薄而稳
- working_memory 要短而新
- lessons 要压缩成可复用结论，不要灌长原文
- samples 比抽象规则更能训练行为

最重要的记忆原则：

- 不要把所有历史都重新灌给每个角色
- 只带当前任务最相关的那部分上下文

## 八、要避免的几件事

### 1. 不要生搬旧角色人格

这个交易项目不应该直接继承 LangGraphLab 那套软件开发角色心理。
复用的是引擎，不是旧 worldview。

### 2. 不要把不确定性都变成治理

角色能安全推进时，就让它推进。
只有继续猜下去成本明显上升时，才升级。

### 3. 不要以为 agent 越多越强

更多 agent = 更多协调成本。
先少角色起步，只有在真实压力反复出现时再扩。

## 九、后续最值得补的文件

一旦项目真正开始推进，下一批最值的文件会是：

- `docs/PROJECT_BLUEPRINT_V1.md`
- `docs/PHASE_BOARD_V1.md`
- `docs/DECISION_LOG_V1.md`

注意：
这些文件应该在真实工作中长出来，不要先用空模板把它们写满。

## 十、短操作清单

在开始一个真实阶段前，先确认：

- 目标是否已经窄到一个阶段能装下
- brief 是否写清了成功标准和风险边界
- 执行角色是否知道什么时候 checkpoint，而不是硬猜
- 新治理是不是因为真实反复痛点才加，不是因为理论上未来可能需要

这份 starter 故意保持轻量。
正确的长法是：
通过真实项目压力逐步长出真正需要的结构，而不是一开始就做成厚重系统。
